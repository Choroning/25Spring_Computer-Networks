## 1. 전달(Delivery)

네트워크 계층은 전달을 관리한다. 패킷을 최종 목적지까지 전달하는 것은 **직접 전달**과 **간접 전달**이라는 두 가지 방법을 사용하여 수행된다.

### 1.1 직접 전달

**직접 전달(Direct Delivery)**은 목적지 호스트가 송신자와 **동일한 물리적 네트워크**에 있을 때 발생한다:

```
         Link
Host A ----------- Host B
    \              /
     \  Switch    /
      +---------+
```

주요 특징:
- 출발지와 목적지가 **동일한 네트워크**에 있음
- **마지막 라우터와 목적지 호스트** 사이에서 수행됨
- 송신자는 목적지 주소에서 **네트워크 주소(netid)**를 추출하여 자신의 네트워크 주소와 비교함
- 일치하면 직접 전달이 수행됨

### 1.2 간접 전달

**간접 전달(Indirect Delivery)**은 목적지 호스트가 **다른 네트워크**에 있을 때 발생한다:

```
  A                                              B
[Host] --- Switch --- Router --- Router --- Switch --- [Host]
  |                     |          |                    |
  +-- Indirect Delivery +          +-- Direct Delivery -+
```

주요 특징:
- 목적지가 송신자와 **동일한 네트워크에 있지 않음**
- 패킷이 **여러 라우터**를 통과함
- 다음 홉 IP 주소를 찾기 위해 **라우팅 테이블**이 사용됨
- 마지막 전달은 항상 직접 전달임

> **핵심 포인트:** 간접 전달에서 송신자와 중간 라우터는 패킷을 다음 라우터로 전달한다. 최종 전달(마지막 라우터에서 목적지 호스트로)만 직접 전달이다.

## 2. 포워딩(Forwarding)

**포워딩**이란 패킷을 목적지까지의 경로에 올려놓는 것을 의미한다. 포워딩을 수행하려면 호스트나 라우터에 **라우팅 테이블**이 있어야 한다.

### 2.1 포워딩 기법

라우팅 테이블과 포워딩 과정을 단순화하는 네 가지 기법이 있다:

**1. 다음 홉 방식(Next-Hop Method):**
- 라우팅 테이블에 완전한 경로 대신 **다음 홉**의 주소만 저장
- 테이블 크기를 줄이고 포워딩 과정을 단순화함

```
Route-Based Table:              Next-Hop Table:
+------+------------------+     +------+---------+
| Dest | Route             |    | Dest | Next Hop|
+------+------------------+     +------+---------+
|  B   | R1, R2, Host B   |    |  B   |   R1    |
+------+------------------+     +------+---------+
```

**2. 네트워크 특정 방식(Network-Specific Method):**
- **모든 목적지 호스트**에 대한 항목 대신 **목적지 네트워크** 하나에 대한 항목만 사용
- 라우팅 테이블 크기를 획기적으로 줄임

```
Host-Specific Table:            Network-Specific Table:
+------+---------+              +------+---------+
| Dest | Next Hop|              | Dest | Next Hop|
+------+---------+              +------+---------+
|  A   |   R1    |              |  N2  |   R1    |
|  B   |   R1    |              +------+---------+
|  C   |   R1    |
|  D   |   R1    |
+------+---------+
```

**3. 호스트 특정 방식(Host-Specific Method):**
- 라우팅 테이블에 **특정 목적지 호스트**에 대한 항목을 포함
- 특정 호스트에 대한 특별한 경로가 필요할 때 사용 (예: 보안이나 특수 처리)
- 네트워크 특정 방식의 역

**4. 기본 경로 방식(Default Method):**
- 모든 네트워크를 나열하는 대신 단일 **기본 항목(0.0.0.0)**을 사용
- 특정 항목과 일치하지 않는 패킷은 기본 라우터로 전송됨

### 2.2 클래스 기반 주소 지정에서의 포워딩

클래스 기반 주소 지정에서 라우터는 세 개의 테이블(Class A, B, C 각각 하나씩)을 가진다.

각 라우팅 테이블 항목은 최소 세 개의 열을 가진다:

| 열 | 설명 |
|--------|-------------|
| Network Address | 목적지 호스트가 위치한 네트워크 |
| Next-Hop Address | 간접 전달을 위해 패킷이 전달되어야 하는 라우터 |
| Interface Number | 나가는 포트 (m0, m1 등) |

**포워딩 과정 (클래스 기반):**
1. 패킷 헤더에서 목적지 IP 주소를 추출
2. 처음 몇 비트에서 클래스(A, B, C)를 결정
3. 클래스별 기본 마스크를 적용하여 네트워크 주소를 추출
4. 해당 테이블에서 네트워크 주소를 검색
5. 발견되면 지정된 인터페이스를 통해 다음 홉 주소로 포워딩
6. 발견되지 않으면 기본 항목을 사용

### 2.3 클래스 없는 주소 지정(CIDR)에서의 포워딩

클래스 없는 주소 지정에서는 마스크와 함께 네트워크 주소(접두사)를 포함하는 **하나의 테이블**이 있다:

```
+------------------+------+-----------+-----------+
| Network/Prefix   | Mask | Next Hop  | Interface |
+------------------+------+-----------+-----------+
| 192.168.1.0/24   | /24  | R1        | m0        |
| 10.0.0.0/8       | /8   | R2        | m1        |
| 0.0.0.0/0        | /0   | R3        | m2        |  <-- Default
+------------------+------+-----------+-----------+
```

**포워딩 과정 (클래스 없는):**
1. 목적지 주소를 추출
2. 테이블의 각 항목에 대해 마스크(AND 연산)를 목적지 주소에 적용
3. 결과가 네트워크 주소와 일치하면 잠재적 매칭임
4. **최장 접두사 매칭**을 적용하여 최적의 경로를 선택
5. 해당하는 다음 홉으로 포워딩

## 3. 라우팅 테이블 구조

### 3.1 라우팅 테이블 열

일반적인 라우팅 테이블은 다음을 포함한다:

| 열 | 설명 |
|--------|-------------|
| Destination Network | 접두사 길이를 포함한 네트워크 주소 |
| Subnet Mask | 목적지 네트워크에 대한 마스크 |
| Next Hop | 다음 라우터의 IP 주소 (또는 "directly connected") |
| Interface | 나가는 인터페이스 식별자 |
| Metric | 목적지까지의 비용/거리 |
| Flags | 경로 유형 표시 (U=up, G=gateway, H=host 등) |

### 3.2 주소 집약

주소 집약(슈퍼네팅)은 라우팅 테이블 항목을 줄인다:

```
Before aggregation:          After aggregation:
192.168.0.0/24  --> R1       192.168.0.0/22 --> R1
192.168.1.0/24  --> R1       (하나의 항목이 네 개를 대체)
192.168.2.0/24  --> R1
192.168.3.0/24  --> R1
```

> **핵심 포인트:** 주소 집약은 네트워크가 연속적이고 동일한 다음 홉을 공유할 때만 작동한다. 이는 인터넷 확장성의 핵심이다.

### 3.3 최장 접두사 매칭

여러 라우팅 테이블 항목이 목적지 주소와 일치할 때, **가장 긴 접두사**(가장 구체적인 매칭)를 가진 항목이 선택된다:

```
Table:
  192.168.0.0/16   --> R1
  192.168.1.0/24   --> R2
  192.168.1.128/25 --> R3

For destination 192.168.1.200:
  /16 matches (192.168.x.x)     --> R1
  /24 matches (192.168.1.x)     --> R2
  /25 matches (192.168.1.128+)  --> R3  <-- 선택됨 (최장 접두사)
```

> **핵심 포인트:** 최장 접두사 매칭은 가장 구체적인 경로가 사용되도록 보장하여, 주소 집약과 특정 예외 경로가 공존할 수 있게 한다.

## 요약

| 개념 | 핵심 포인트 |
|---------|-----------|
| 직접 전달 | 출발지와 목적지가 동일 네트워크; 마지막 홉은 항상 직접 전달 |
| 간접 전달 | 패킷이 라우터를 통과; 다음 홉을 위해 라우팅 테이블 사용 |
| 다음 홉 방식 | 완전한 경로가 아닌 다음 라우터만 저장 |
| 네트워크 특정 | 호스트별이 아닌 네트워크별로 하나의 항목 |
| 기본 경로 | 알 수 없는 목적지를 위한 포괄 항목 (0.0.0.0/0) |
| 클래스 기반 포워딩 | 클래스별 별도 테이블; 클래스 마스크로 netid 추출 |
| 클래스 없는 포워딩 | 접두사/마스크가 있는 단일 테이블; 매칭을 위한 AND 연산 |
| 최장 접두사 매칭 | 가장 구체적인(가장 긴 접두사) 매칭 항목이 우선 |
| 주소 집약 | 연속된 네트워크를 결합하여 테이블 크기 감소 |

## 부록

### A. 예제: 클래스 기반 포워딩

목적지 IP: 192.168.5.100이 주어진 경우

1. 처음 3비트가 `110` --> Class C
2. 기본 마스크: 255.255.255.0 (/24)
3. 네트워크 주소: 192.168.5.0
4. Class C 라우팅 테이블에서 192.168.5.0 검색
5. 발견되면 지시된 다음 홉으로 포워딩
6. 발견되지 않으면 기본 경로 사용

### B. 예제: 클래스 없는 포워딩

목적지 IP: 10.1.2.100과 라우팅 테이블이 주어진 경우:

| Network | Mask | Next Hop |
|---------|------|----------|
| 10.1.0.0 | /16 | R1 |
| 10.1.2.0 | /24 | R2 |
| 0.0.0.0 | /0 | R3 |

1. 10.1.2.100 AND /16 = 10.1.0.0 --> 일치 (R1)
2. 10.1.2.100 AND /24 = 10.1.2.0 --> 일치 (R2)
3. 10.1.2.100 AND /0 = 0.0.0.0 --> 일치 (R3)
4. 최장 접두사 매칭: /24 --> **R2**로 포워딩

### C. 실제 라우팅 테이블 (Linux)

```
$ ip route show
default via 192.168.1.1 dev eth0
10.0.0.0/8 via 10.0.0.1 dev eth1
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.100
```
